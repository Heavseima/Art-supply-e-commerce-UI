This repo runs Next.js **16.2.7**, React **19.2.4**, Tailwind **v4** (`@tailwindcss/postcss`). Package manager: npm. App Router under `src/app`. Path alias `@/*` -> `./src/*` (tsconfig).

Verified from `node_modules/next/dist/docs/` (read these, do not assume from training data):

**Caching model = "Cache Components"** (the big change vs older Next):
- Enable with `cacheComponents: true` in `next.config.ts`. This REPLACES the old `experimental.ppr`, `experimental.dynamicIO`, `experimental.useCache` (all removed). PPR becomes the default rendering behavior.
- Caching API is the **`'use cache'` directive** (string literal at top of file/component/function) — NOT `unstable_cache`. Pair with `cacheLife('hours'|...)` and `cacheTag('tag')` imported from `next/cache`.
- `fetch` is NOT cached by default and blocks render. Opt in via `'use cache'`, or wrap fetching component in `<Suspense>` to stream.
- Uncached async data accessed outside `<Suspense>` or `'use cache'` => build/dev error "Uncached data was accessed outside of <Suspense>" (blocking-route). So every dynamic data read must be either cached or inside Suspense.
- Closure vars + args of a `'use cache'` fn become part of the cache key automatically.
- `React.cache()` (from 'react') still exists for per-request memoization.

**Routing / props:**
- `params` and `searchParams` are **Promises** — must be `await`ed in async components.
- Global generated type helpers exist: `PageProps<'/route/[slug]'>` and `LayoutProps<'/route'>` — no import needed, generated by `next dev`/`next build`/`next typegen`. params resolve to `{}` for static routes.
- `generateStaticParams()` replaces getStaticPaths; returns `{ slug: string }[]`.

**Gotcha confirmed at build time:** awaiting `searchParams`/`params` (or any uncached async read) at the TOP LEVEL of a page component (outside Suspense) fails the prerender with "Uncached data was accessed outside of <Suspense>" / blocking-route. Fix: push the dynamic read into a child component rendered INSIDE a `<Suspense>` boundary; pass the `searchParams`/`params` Promise down as a prop. Verified working for `/products` -> shows as ◐ Partial Prerender in build output. `generateStaticParams` reads ARE fine outside Suspense.

**Also:** `next typegen` (or `next dev`/`next build`) must run before `tsc --noEmit` will accept `PageProps<'/route'>` for any non-`/` route — the literal route union is generated. Non-deterministic ops like `new Date()` in a prerendered (static-shell) component also error; use a constant or defer via `connection()`+Suspense.

**Client Router Cache + component state:** A `<Link>`/`Button href` client navigation reuses cached route segments, so a client component's local React state (`useState`) can SURVIVE leaving and returning to a route — it is NOT guaranteed to remount fresh. Docs (instant-navigation.md): on a client nav between siblings only the segment below the shared layout re-renders; tree above is reused. Implication: don't rely on "navigating back resets my component". Derive view state from the source of truth (e.g. the LocalStorage cart) instead of a stale local flag. ESLint rule `react-hooks/set-state-in-effect` also FORBIDS synchronous `setState` inside `useEffect` — prefer deriving values during render over reset-effects.

**Why:** AGENTS.md/CLAUDE.md explicitly warn the API differs from training data.
**How to apply:** Use `'use cache'`+`cacheLife`/`cacheTag` for explicit caching; await params/searchParams INSIDE a Suspense child; run `next typegen` before type-checking.
